SPRINT 25 — DeepEval adoption + open-set document-type
classifier. 4 tasks + verification. Do NOT expand scope beyond
what's listed — no OCR/ingestion pipeline (that's Chancery, S26),
no retrieval/embeddings (also S26+). This sprint is: adopt
DeepEval, build the classifier, build the eval harness, expose
its model choice in the admin settings UI.

CONTEXT: reference_data already has a seeded 'doc_category' list
(12 values, from Sprint 17/19-era work) — READ ITS LIVE SCHEMA
AND ACTUAL VALUES FIRST (do not guess). org_settings already has
ai.model.* keys (Mini-Bedrock sprint), including a precedent for
TASK-SPECIFIC model overrides — the existing 'assistant' task
already resolves to a different model (Sonnet) than
ai.model.default (Haiku). READ HOW THAT EXISTING TASK-SPECIFIC
OVERRIDE IS KEYED/STRUCTURED before adding a new one — follow the
same convention, do not invent a different pattern.

IMPORTANT FRAMING: "any document type" means this is an OPEN-SET
classifier, not a fixed enum. It should:
  (a) try to match the document against the EXISTING doc_category
      reference list first
  (b) if nothing fits well, PROPOSE a new category rather than
      forcing a bad match or inventing one silently — proposals
      go into a review queue, they do NOT get auto-inserted into
      the canonical reference_data list. A human ratifies new
      categories. This mirrors the existing "AI proposes, human
      ratifies into a fixed versioned schema" pattern already
      used elsewhere in this codebase — do not silently expand
      the canonical list; do not force-fit either.

STANDING RULES: org_id never from request body; Decimal for
any money fields; no interactive prompts; light theme if any UI
is touched.

=== TASK 1: DeepEval adoption ===
Install DeepEval (Apache 2.0). Set up base config. IMPORTANT:
DeepEval's LLM-judge-based metrics default to OpenAI — but the
metric this sprint needs (document-type sort accuracy) is a
NO-JUDGE metric (exact/category match against a known label),
so no judge model is actually needed for THIS metric. Still,
document this trap clearly in code comments for whoever adds a
judge-needed metric later (e.g. retrieval faithfulness in S26) —
that future work MUST set the judge model explicitly via the
existing ai.model.* org_settings config, never let it default.
Build a custom DeepEval metric class (extends BaseMetric):
DocumentTypeSortAccuracy — scores whether the classifier's
returned category matches the expected/known category for a
test case.

=== TASK 2: Open-set document-type classifier ===
First: read the live reference_data table WHERE list_key =
'doc_category' (or whatever the actual list_key is — check,
don't assume) — get its real columns and current 12 seeded
values.
Build a classifier service (e.g.
apps/api/services/document_classifier.py):
  - Input: document text (plain text — assume already extracted;
    OCR/ingestion itself is out of scope, Chancery's job later)
  - Prompts the model with the full list of existing doc_category
    values as candidates, PLUS the option to propose something
    new if nothing fits
  - Returns: { category_code, confidence, is_new_proposal: bool,
    reasoning }
  - If is_new_proposal is true, INSERT into a review queue (design
    the minimal schema for this — a table or a status column on
    a new proposal record; keep it simple, matching the
    reference_data house pattern where sensible) rather than
    reference_data directly. Never auto-insert a new canonical
    category.
  - MODEL SELECTION: add a new task-specific org_settings key
    (e.g. ai.model.document_classifier) following the SAME
    naming/structure convention as the existing 'assistant'
    task-type override found above. Seed its default value to
    the current Haiku-class model string (confirm the exact
    string from ai.model.default, which Mini-Bedrock found
    already resolves to a Haiku model — this classifier's
    default IS ai.model.default's value, effectively, unless a
    dedicated key is cleaner given the existing pattern — use
    whichever approach matches how 'assistant' was already done).
    The classifier resolves its model by checking this
    task-specific key first, falling back to ai.model.default if
    the org hasn't explicitly overridden it. This lets an
    org_admin choose a stronger model for their own classifier
    while every org gets Haiku by default.

=== TASK 3: Eval harness + placeholder run ===
Write apps/api/scripts/eval_document_classifier.py using
DeepEval's test-case format:
  - Accepts a directory or fixture list of {text, expected_category}
    test cases — design this so dropping in REAL documents later
    requires no code changes, just adding fixtures
  - Since NO real documents exist yet: generate a SMALL set (8-10)
    of clearly-synthetic placeholder text snippets (short,
    obviously-fake, spanning a few different document flavors —
    e.g. a fake K-1-style snippet, a fake correspondence snippet,
    a fake estate-plan-style snippet), each with a known expected
    category
  - Run the classifier against these placeholder cases, score with
    DocumentTypeSortAccuracy
  - The report MUST print an unmissable banner stating this is a
    PLACEHOLDER/SYNTHETIC run — not representative of real-world
    accuracy — and that the real evaluation requires real
    documents to be dropped into the fixture set later
